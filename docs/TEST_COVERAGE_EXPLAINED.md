# 테스트 커버리지 쉽게 설명하기

## 🎯 테스트 커버리지란?

**테스트 커버리지 = 코드 중에서 테스트가 얼마나 많은 부분을 검증했는지**

### 비유로 설명하면

**🏥 건강검진 비유:**
- 전체 신체 검사 = 100% 커버리지
- 일부만 검사 = 낮은 커버리지
- 테스트 커버리지 = "코드의 몇 %를 테스트했나?"

**📚 책 읽기 비유:**
- 책 전체를 읽음 = 100% 커버리지
- 일부만 읽음 = 낮은 커버리지
- 테스트 커버리지 = "코드의 몇 %를 테스트했나?"

---

## 📊 커버리지 계산 방법

### 예시: 간단한 함수

```python
def add(a, b):
    if a < 0:
        return 0
    return a + b
```

**테스트 케이스 1:**
```python
def test_add():
    result = add(2, 3)
    assert result == 5
```

**커버리지 분석:**
- ✅ `return a + b` 실행됨 (테스트됨)
- ❌ `if a < 0:` 실행 안 됨 (테스트 안 됨)
- **커버리지: 50%** (2줄 중 1줄만 테스트)

**테스트 케이스 2 (추가):**
```python
def test_add_negative():
    result = add(-1, 3)
    assert result == 0
```

**커버리지 분석:**
- ✅ `if a < 0:` 실행됨 (테스트됨)
- ✅ `return 0` 실행됨 (테스트됨)
- ✅ `return a + b` 실행됨 (테스트됨)
- **커버리지: 100%** (모든 줄 테스트됨)

## 📊 커버리지 종류

### 1. 라인 커버리지 (Line Coverage)
**의미:** 코드의 몇 % 줄이 실행되었나?

**예시:**
```python
def process_data(data):
    if data is None:      # 라인 1
        return None       # 라인 2
    result = data * 2     # 라인 3
    return result         # 라인 4
```

**테스트:**
```python
def test_process_data():
    result = process_data(5)
    assert result == 10
```

**커버리지:**
- 라인 1: ✅ 실행됨
- 라인 2: ❌ 실행 안 됨 (data가 None이 아니므로)
- 라인 3: ✅ 실행됨
- 라인 4: ✅ 실행됨
- **라인 커버리지: 75%** (4줄 중 3줄)

---

### 2. 브랜치 커버리지 (Branch Coverage)
**의미:** if/else 등 분기문의 몇 %가 실행되었나?

**예시:**
```python
def check_status(status):
    if status == "active":    # 브랜치 1
        return "OK"
    elif status == "pending": # 브랜치 2
        return "WAIT"
    else:                     # 브랜치 3
        return "ERROR"
```

**테스트:**
```python
def test_check_status():
    assert check_status("active") == "OK"
    # 브랜치 1만 테스트됨
```

**커버리지:**
- 브랜치 1: ✅ 테스트됨
- 브랜치 2: ❌ 테스트 안 됨
- 브랜치 3: ❌ 테스트 안 됨
- **브랜치 커버리지: 33%** (3개 중 1개)

---

### 3. 함수 커버리지 (Function Coverage)
**의미:** 함수의 몇 %가 호출되었나?

**예시:**
```python
def function_a():  # 함수 1
    pass

def function_b():  # 함수 2
    pass

def function_c():  # 함수 3
    pass
```

**테스트:**
```python
def test_functions():
    function_a()  # 함수 1만 호출
```

**커버리지:**
- 함수 1: ✅ 호출됨
- 함수 2: ❌ 호출 안 됨
- 함수 3: ❌ 호출 안 됨
- **함수 커버리지: 33%** (3개 중 1개)

**도구:** `pytest-cov` (Python 테스트 커버리지 도구)

**목표:** 60% 이상

**확인 방법:**
```bash

```

**결과:**

- **전체: 60% 커버리지** (목표 달성)

## 💡 왜 중요한가?

### 높은 커버리지의 장점

1. **버그 조기 발견**
   - 테스트되지 않은 코드에 버그가 있을 수 있음
   - 높은 커버리지 = 더 많은 버그 발견

2. **리팩토링 안전성**
   - 코드를 수정해도 테스트로 검증 가능
   - 높은 커버리지 = 안전한 리팩토링

3. **문서화 효과**
   - 테스트 코드가 사용 예시 역할
   - 높은 커버리지 = 더 많은 예시

### 낮은 커버리지의 위험

1. **숨겨진 버그**
   - 테스트되지 않은 코드에 버그가 있을 수 있음
   - 프로덕션에서 발견될 수 있음

2. **리팩토링 위험**
   - 코드를 수정해도 테스트가 없어 검증 불가
   - 실수로 기능이 깨질 수 있음

---

## 🎯 우리 프로젝트의 목표

### 현재 목표: 60% 이상

**의미:**
- 코드의 60% 이상이 테스트로 검증됨
- 나머지 40%는 테스트되지 않음

**왜 60%인가?**
- 너무 높으면 (90%+) 테스트 작성 시간이 많이 걸림
- 너무 낮으면 (30% 이하) 버그 발견이 어려움
- 60%는 **균형잡힌 목표**

## 📊 커버리지 확인 방법

### 1. 로컬에서 확인

```bash
cd backend
pytest tests/ --cov=app --cov-report=html
```

**결과:**
- 터미널에 커버리지 % 표시
- `htmlcov/index.html` 파일 생성

**HTML 리포트 열기:**
```bash
# Windows
start htmlcov/index.html

# Mac/Linux
open htmlcov/index.html
```

### 2. CI에서 확인

**GitHub Actions에서 자동 확인:**
- PR 생성 시 자동으로 커버리지 확인
- 60% 미만이면 경고 (또는 실패)

## 💡 커버리지 향상 방법

### 1. 테스트 케이스 추가

**낮은 커버리지 파일:**
```python
# app/services/quality.py (50% 커버리지)
```

**개선 방법:**
- 테스트되지 않은 함수에 테스트 추가
- 엣지 케이스 테스트 추가
- 에러 케이스 테스트 추가

### 2. 불필요한 코드 제거

**사용되지 않는 코드:**
```python
def unused_function():  # 테스트 안 됨
    pass
```

**개선 방법:**
- 사용되지 않는 코드 제거
- 또는 테스트 추가
