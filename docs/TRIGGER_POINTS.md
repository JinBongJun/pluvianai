# 아키텍처 전환 트리거 포인트

이 문서는 AgentGuard의 아키텍처 변경이 필요한 시점을 정의합니다.

## 트리거 포인트 정의

### 1. Agent 서비스 분리

**트리거**: 월 API 호출 10만+ 건

**조치**:
- Agent 서비스를 별도 서비스로 분리
- Railway에서 독립적인 서비스로 배포
- API Service와 Agent Service 간 통신 최적화

**이유**:
- Agent 실행은 CPU/메모리 집약적
- 독립적인 스케일링 필요
- API 서비스와의 리소스 경합 방지

### 2. Supabase Auth 검토

**트리거**: 사용자 1,000+ 명

**조치**:
- Supabase Auth로 전환 검토
- 현재 인증 시스템과 비교 분석
- 마이그레이션 계획 수립

**이유**:
- 사용자 관리 복잡도 증가
- 인증/인가 로직 표준화
- 보안 강화

### 3. 비용 최적화 강화

**트리거**: Agent 비용 $500+/월

**조치**:
- LLM API 호출 최적화
- 캐싱 전략 강화
- 모델 선택 최적화

**이유**:
- 비용 효율성 개선
- 사용자 비용 절감
- 경쟁력 강화

### 4. 데이터베이스 스케일링

**트리거**: 
- 데이터베이스 크기 10GB+
- 쿼리 응답 시간 1초+ (평균)

**조치**:
- 데이터베이스 인덱스 최적화
- 아카이빙 전략 강화
- 읽기 전용 복제본 검토

**이유**:
- 성능 유지
- 사용자 경험 개선
- 확장성 확보

### 5. Supabase 전환

**트리거**: 
- 데이터베이스 관리 복잡도 증가
- 인증/스토리지 기능 필요성 증가

**조치**:
- SupabaseRepository 구현
- 점진적 마이그레이션
- 기존 기능 호환성 유지

**이유**:
- 운영 부담 감소
- 기능 확장 용이
- 비용 효율성

## 모니터링 지표

다음 지표를 지속적으로 모니터링하여 트리거 포인트를 감지합니다:

- 월 API 호출 수
- 활성 사용자 수
- Agent 실행 비용
- 데이터베이스 크기
- 평균 쿼리 응답 시간
- 서버 리소스 사용률

## 트리거 포인트 대응 프로세스

1. **감지**: 모니터링 지표 확인
2. **평가**: 트리거 포인트 도달 여부 확인
3. **계획**: 아키텍처 변경 계획 수립
4. **검토**: 기술적/비즈니스적 타당성 검토
5. **구현**: 단계적 구현 및 테스트
6. **모니터링**: 변경 후 지표 모니터링
