# 모니터링 시스템 쉽게 설명하기

지금까지 만든 모니터링 시스템이 무엇인지 쉽게 설명합니다.

## 🎯 모니터링이란?

**모니터링 = 서비스가 잘 돌아가고 있는지 지켜보는 것**

예를 들어:
- 🏥 병원의 심박수 모니터처럼
- 🚗 자동차의 계기판처럼
- 📊 회사의 매출 대시보드처럼

---

## 📊 우리가 만든 모니터링 시스템

### 1️⃣ **백엔드 메트릭 수집 시스템** (Prometheus)

**뭐하는 거?**
- 백엔드가 얼마나 바쁜지, 얼마나 빠른지 측정
- API 요청 수, 응답 시간, 에러 수 등을 자동으로 수집

**어디서 확인?**
```
https://your-backend.railway.app/metrics
```

**수집하는 데이터:**
- ✅ 총 API 요청 수
- ✅ API 응답 시간 (얼마나 빠른지)
- ✅ 에러 발생 수
- ✅ 데이터베이스 쿼리 수
- ✅ LLM API 호출 수
- ✅ LLM API 비용
- ✅ 활성 사용자 수
- ✅ 활성 프로젝트 수

**비유로 설명하면:**
- 백엔드의 "건강 체크" 시스템
- 매초마다 "심박수", "체온", "혈압"을 측정하는 것

---

### 2️⃣ **Grafana 대시보드** (로컬용, 선택사항)

**뭐하는 거?**
- Prometheus가 수집한 데이터를 예쁜 그래프로 보여주는 대시보드
- 여러 지표를 한 화면에 모아서 보기

**어디서 확인?**
- 로컬에서만 실행 가능 (Docker Compose 사용)
- `http://localhost:3001` (로컬 개발용)

**보여주는 것:**
- 📈 API 요청 수 그래프
- ⏱️ 응답 시간 그래프
- 💰 LLM API 비용 그래프
- 📊 품질 점수 분포
- 🚨 에러 발생 추이

**비유로 설명하면:**
- 병원의 모니터 화면처럼 여러 지표를 한 번에 보는 것
- 하지만 **프로덕션에서는 Railway Metrics를 사용**하므로 선택사항

---

### 3️⃣ **Vercel Analytics** (프론트엔드)

**뭐하는 거?**
- 프론트엔드가 얼마나 많이 사용되는지, 얼마나 빠른지 측정
- 사용자가 어떤 페이지를 많이 보는지 추적

**어디서 확인?**
- Vercel 대시보드 → 프로젝트 → **Analytics** 탭
- https://vercel.com/dashboard

**수집하는 데이터:**
- ✅ 페이지뷰 수 (얼마나 많은 사람이 방문했는지)
- ✅ 고유 방문자 수
- ✅ 페이지 로드 시간 (얼마나 빠른지)
- ✅ 에러 발생 수
- ✅ 사용자 세션 분석

**비유로 설명하면:**
- 웹사이트의 "방문자 통계" 시스템
- Google Analytics 같은 것

---

### 4️⃣ **Railway Metrics** (백엔드)

**뭐하는 거?**
- Railway가 백엔드 서버의 상태를 자동으로 측정
- CPU, 메모리, 네트워크 사용량을 실시간으로 보여줌

**어디서 확인?**
- Railway 대시보드 → 프로젝트 → Backend 서비스 → **Metrics** 탭
- https://railway.com/dashboard

**수집하는 데이터:**
- ✅ CPU 사용률 (%)
- ✅ 메모리 사용량 (MB)
- ✅ 네트워크 트래픽 (MB/s)
- ✅ 디스크 사용량

**비유로 설명하면:**
- 서버의 "리소스 사용량" 모니터
- 컴퓨터 작업 관리자처럼 CPU/메모리 사용량 보는 것

---

### 5️⃣ **Railway Logs** (백엔드)

**뭐하는 거?**
- 백엔드가 실행되면서 출력하는 모든 로그를 실시간으로 보여줌
- 에러가 발생하면 여기서 확인 가능

**어디서 확인?**
- Railway 대시보드 → 프로젝트 → Backend 서비스 → **Logs** 탭
- https://railway.com/dashboard

**보여주는 것:**
- ✅ API 요청 로그
- ✅ 에러 메시지
- ✅ 데이터베이스 쿼리 로그
- ✅ 디버그 정보

**비유로 설명하면:**
- 서버의 "일기장" 또는 "기록부"
- 문제가 생기면 여기서 원인을 찾을 수 있음

---

### 6️⃣ **Sentry** (에러 추적)

**뭐하는 거?**
- 에러가 발생하면 자동으로 알려주고 상세 정보를 수집
- 어디서, 언제, 왜 에러가 발생했는지 추적

**어디서 확인?**
- Sentry 대시보드: https://sentry.io
- 프로젝트 → **Issues** 탭

**수집하는 데이터:**
- ✅ 에러 발생 위치 (어느 파일, 어느 줄)
- ✅ 에러 발생 시점
- ✅ 에러 발생 빈도
- ✅ 스택 트레이스 (에러가 어떻게 발생했는지)
- ✅ 사용자 컨텍스트 (누가 에러를 겪었는지)

**비유로 설명하면:**
- 자동 "에러 감지 및 알림" 시스템
- 문제가 생기면 즉시 알려주는 경보 시스템

## 🎯 실제로 어떻게 사용하나?

### 일상적인 확인 (매일 아침)

1. **Vercel Analytics** 확인
   - "어제 몇 명이 방문했나?"
   - "어떤 페이지가 인기인가?"

2. **Railway Metrics** 확인
   - "서버가 정상적으로 돌아가고 있나?"
   - "CPU/메모리가 너무 많이 쓰이고 있나?"

3. **Sentry** 확인
   - "새로운 에러가 발생했나?"
   - "에러가 많이 발생했나?"

### 문제 발생 시

1. **Railway Logs** 확인
   - "에러 메시지가 뭐지?"
   - "어떤 요청에서 문제가 생겼지?"

2. **백엔드 메트릭** 확인
   - "응답 시간이 느려졌나?"
   - "에러율이 높아졌나?"

3. **Vercel Analytics** 확인
   - "프론트엔드에서도 문제가 있나?"
   - "어떤 페이지에서 에러가 발생했나?"

## 📊 모니터링 시스템 비교

| 시스템 | 용도 | 어디서 확인 | 자동/수동 |
|--------|------|------------|----------|
| **백엔드 메트릭** | 백엔드 성능 측정 | `/metrics` 엔드포인트 | ✅ 자동 |
| **Grafana** | 메트릭 시각화 (로컬) | `localhost:3001` | ⚠️ 수동 설정 필요 |
| **Vercel Analytics** | 프론트엔드 분석 | Vercel 대시보드 | ✅ 자동 |
| **Railway Metrics** | 서버 리소스 모니터링 | Railway 대시보드 | ✅ 자동 |
| **Railway Logs** | 백엔드 로그 확인 | Railway 대시보드 | ✅ 자동 |
| **Sentry** | 에러 추적 및 알림 | Sentry 대시보드 | ✅ 자동 |

---

## 💡 핵심 정리

### 프로덕션에서 자주 사용하는 것 (추천 ⭐)

1. **Vercel Analytics**: 프론트엔드 사용량 확인
2. **Railway Metrics**: 백엔드 리소스 확인
3. **Railway Logs**: 에러 확인
4. **Sentry**: 에러 추적 및 알림

### 로컬 개발용 (선택사항)

1. **Grafana**: 상세한 메트릭 대시보드 (로컬에서만)
2. **Prometheus**: 메트릭 수집 (백엔드에 내장됨)

## 🎯 실제 사용 예시

### 시나리오 1: "서비스가 느려졌어요"

**확인 순서:**
1. Railway Metrics → CPU/메모리 확인
2. Railway Logs → 에러 로그 확인
3. 백엔드 메트릭 (`/metrics`) → 응답 시간 확인
4. Vercel Analytics → 프론트엔드 성능 확인

### 시나리오 2: "에러가 발생했어요"

**확인 순서:**
1. Sentry → 에러 상세 정보 확인
2. Railway Logs → 에러 로그 확인
3. 백엔드 메트릭 → 에러율 확인

### 시나리오 3: "사용자가 얼마나 있나요?"

**확인 순서:**
1. Vercel Analytics → 방문자 수 확인
2. 백엔드 메트릭 → 활성 사용자 수 확인
