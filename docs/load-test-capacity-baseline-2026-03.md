# Load Test Capacity Baseline (2026-03)

이 문서는 Live View + ingest 혼합 부하 기준의 용량 검증 결과를 기록하고, 운영선(보수/공격)과 상단 경계를 명시하기 위한 기준 문서다.

---

## 1) 목표와 범위

- 목표: `10,000` 사용자 규모를 가정했을 때 안정 운영 가능한 동시 부하 구간 확인
- 범위: `live_view` 폴링 + `ingest` 백그라운드 유입 혼합 시나리오
- 제외: Release Gate 혼합 부하는 본 문서 범위 밖 (별도 시나리오로 관리)

---

## 2) 테스트 환경

- 대상 API: `https://api.pluvianai.com`
- 프로젝트: `project_id=5`
- 부하 도구: `k6 v1.7.0 (windows-amd64)`
- 트래픽 형태:
  - `live_view`: `/api/v1/projects/{project_id}/live-view/agents?limit=30&include_deleted=1&compact=1`
  - `ingest`: `/api/v1/projects/{project_id}/api-calls`
- VU sleep:
  - live_view: `1.0s ~ 2.5s`
  - ingest: `0.8s ~ 1.2s`

---

## 3) 운영 SLO 기준 (현재 합의안)

- 가용성: `http_req_failed < 1%` (운영선), `< 2%` (공격 운영선 상한)
- 지연: `http_req_duration p95 < 7s` (운영선), `< 10s` (상단 경계 판단선)
- 보조 지표:
  - 429가 주 실패 원인인지 여부
  - interrupted iterations 발생 여부
  - 장기 드리프트(10분 -> 20분 -> 30분) 유무

## 5) 결론 (2026-03 기준)

- 보수 운영선: `160 live_view + 20 ingest`
- 공격 운영선: `180 live_view + 20 ingest`
- 상단 경계(천장 근처): `200 live_view + 20 ingest`

해석:
- 현재 기준에서 `10,000 사용자 규모` 목표를 위한 혼합 부하 운영 근거는 충분하다.
- 단, `10,000명 = 동시접속 10,000명`은 아니므로 제품 사용 패턴(동시성, poll 주기, ingest 유입률)에 따라 운영선은 재평가한다.

## 6) 다음 액션

- 운영 기준 고정:
  - 기본 운영선: `160+20`
  - 이벤트성 피크/공격 운영선: `180+20`
- 정기 재검증:
  - 배포 후: `160+20 / 10m` 스모크
  - 주간/격주: `160+20 / 30m` soak
  - 주요 변경(캐시, DB 인덱스, 이벤트 파이프라인) 시: `180+20 / 30m` 재검증
- 알림 조건(권장):
  - `http_req_failed >= 1%` 5분 지속
  - `p95 >= 7s` 5분 지속
  - `429` 비중 급증 시 rate-limit vs backend saturation 분리 진단
