# 2026.04 ~ 2026.12 컨텐츠 캘린더 (주 2회, 레딧 + IH)

## 0) 전제
- 기간: `2026-04-01` ~ `2026-12-31` (주 2회 업로드 기준)
- 채널 리듬: 매주 `Reddit(질문/토론)` 1편 + `Indie Hackers(IH, 상황/회고)` 1편
- 목표 포지셔닝: “설정 바꾸고 무턱대고 배포할 때 생기는 회귀/불안정을 배포 전에 조기에 캐치”가 메인
- 제품명: 서비스명은 `PluvianAI`만 사용 (레포 폴더명/“AgentGuard” 류는 대외에서 생략)
- 용어 엄수: “골든 데이터셋” 같은 공식명칭은 쓰지 않고, `저장해둔 케이스/저장된 스냅샷/저장된 실제 트래픽`처럼 풀어 씀
- 홍보 금지 톤: “클릭해서 설정하세요 / 임계값 입력법” 같은 매뉴얼 톤은 금지. 반드시 본문은 `상황 → (왜 문제 생김) → 조기 캐치가 왜 되는지(한 문장) → 질문/결론` 구조

## 1) 포스트 템플릿 (Reddit / IH 공통)
### 1.1 Reddit(질문형) 템플릿
- [상황 1문장] “모델/프롬프트/벤더를 바꾼 뒤 평균은 괜찮은데 엣지에서만 조용히 깨짐”
- [고통 1문장] “수동 테스트(10~20개)는 대충 통과하는데, 배포 후에만 발견”
- [핵심 질문 1개] “여러분은 이런 ‘묻히는 회귀’를 어떻게 잡나요?”
- [마이크로 개입 1~2문장] “우리는 저장해둔 실제 케이스를 후보 설정에 다시 돌려서 케이스 단위로 먼저 확인하는 쪽으로 접근 중”
  - 제품 용어(Release Gate/Clinical Log)는 마지막 문장에만 최소로 언급
- [후속 질문(선택)] “추천하는 최소 테스트 세트 크기/구성은요?”

### 1.2 IH(상황/회고형) 템플릿
- [도입 1~2문장] “무슨 변경을 했고, 어떤 일이 생겼는지(익명화 필수)”
- [실패 모드 1개] “평균만 보면 안 보이던 케이스가 있었다 / 플레이크가 숨어 있었다 / 오케스트레이션이 꼬였다”
- [우리의 레버 1개] “배포 전 같은 케이스를 후보 설정으로 재실행해서 먼저 확인”
  - “Live View는 재료 수집, 메인은 배포 전에 재실행/판정”을 1줄로 정리
- [결과 1개] “어느 유형이 미리 잡혔는지(실패 1~2건 정도로만)”
- [트레이드오프 1개] “비용/반복 횟수/키 운영 때문에 전부를 매번 돌리진 않는다”
- [마무리 1문장 CTA] “여러분은 배포 전 어떤 루틴으로 ‘조용한 회귀’를 놓치지 않나요?”

## 2) 스크린샷(화면) 사용 규칙
- 목적: “설정법”을 가르치기 위함이 아니라 “증거(결과)가 어떻게 보이느냐”를 보여주기 위함
- Shot 번호는 `docs/demo/content-plan-support-bot-screenshots.md`의 정의를 그대로 사용
- 추천 비율
  - Reddit: `Shot 1(훅)` + `Shot 3(평가 결과)` 조합이 가장 무난
  - IH: `Shot 2b(재료가 쌓임)` 또는 `Shot 3(평가)` + 결론 `Shot 6(실패가 잡힌 run)` 중심
- 크롭 규칙(문서에 이미 추가된 규칙 준수)
  - “오버라이드 입력/JSON” 화면(Shot 5)은 길게 보이지 않게 잘라서 “Run 전에 다른 설정을 넣었다”만 보이게
  - 결과(Shot 3/6)는 더 크게 잡기

## 3) 캘린더 구조
- 한 달을 기본 `4주(=8편)`으로 설계합니다. 실제 달에 5주가 있으면 **추가 슬롯은 바로 아래 ‘추가 주차 슬롯’ 템플릿**을 복제해서 채우면 됩니다.
- 이 문서는 “주차 단위의 시나리오 + 포스트 제목 방향 + 스크린샷 선택”까지 포함합니다.
