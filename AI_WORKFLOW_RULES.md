# AI Workflow Rules (AgentGuard 기준)

## 0. 핵심 원칙
- AI는 “주니어 엔지니어”처럼 다룬다: 명확한 요구사항/문서 없이 작업 금지
- 만들 수 있는 것 ≠ 팔리는 것: “내가 매일 쓰는 기능”이 우선순위
- 범위는 무조건 제한한다: 요구사항 추가는 구조 붕괴의 시작

---

## 0-1) 빠른 적용 10줄 요약
1. Plan 없이 구현 금지
2. 한 작업 = 한 마일스톤 = 한 커밋
3. 결제/보안/모니터링 먼저 설계
4. 범위 추가는 비목표로 분리
5. P95/에러율/비용 지표 정의
6. 로컬보다 클라우드 기준 측정
7. 검증 1개는 반드시 실행
8. 구조 취약하면 리팩터 후보에 기록
9. 반복 작업은 커맨드화
10. 변경 후 요약/리스크/다음 작업 기록

---

## 1) 문서화 구조 (필수)
1. Tech Spec (무엇/왜/하지 않을 것)
2. Plan.md (오늘 할 일, 마일스톤 단위)
3. 완료 후 체크/커밋/검증 기록

규칙:
- Plan 없이 구현 금지
- 한 작업 = 한 마일스톤 = 한 커밋
- 작은 변경으로 쪼개기

---

## 2) AI 협업 방식
- “이게 최선인가?”를 항상 질문
- 모델 간 충돌 유도 → 합의 도출
- 계획 확정 후 실행, 검증 필수

---

## 2-1) 작업 단계 (권장 루틴)
1. 문제 정의: 사용자/사업 관점에서 한 문장
2. Tech Spec: 목표/비목표/요구사항/성공 기준
3. Plan: 3~7개 마일스톤으로 쪼개기
4. 구현: 한 마일스톤만 처리
5. 검증: 테스트/수동 확인/로그 중 하나
6. 기록: Plan 체크 + 커밋

---

## 3) 결제/구독은 기능보다 어렵다
필수 설계 항목:
- 신규 결제 → 활성화
- 업그레이드/다운그레이드
- 취소/재구독
- 환불/중복결제/결제 실패

규칙:
- “결제 설계 먼저, 기능은 나중”
- 관리형 결제(Stripe/LemonSqueezy/Paddle) 우선

---

## 3-1) 결제 실패 케이스 예시 (반드시 체크)
- 결제 성공 후 다른 계정에 크레딧 반영
- 취소 후 재구독 시 권한/기간 꼬임
- 업그레이드 직후 다운그레이드 반복
- 환불 처리 후 접근권한 유지

---

## 4) 운영 필수 5종 세트
- 에러 로깅 (Sentry)
- 사용자 행동 분석 (PostHog/GA)
- 백업/복구 전략
- 상태 페이지/헬스 체크
- 관리자 화면 (유저/결제/콘텐츠 확인)

---

## 5) 보안 체크리스트 (PR 기준 최소 통과)
- CORS/Preflight, CSRF
- XSS/CSP, SSRF 방어
- AuthN/AuthZ, RBAC/ABAC, 최소권한
- Validation + SQLi 방어
- RateLimit/Bruteforce
- Cookie(HttpOnly/Secure/SameSite)
- Secret 관리/Rotation
- HTTPS/HSTS
- Audit Log
- 에러 노출 차단
- 의존성 취약점 점검

---

## 5-1) 배포 전 필수 체크
- 환경변수 누락 없는지
- CORS/도메인 설정 확인
- HTTPS/리다이렉션 정상 여부
- DB 마이그레이션 반영 여부
- 롤백/백업 준비 여부

---

## 6) 모니터링/수치화
- 병목은 가장 좁은 구간에서 발생
- 핵심 지표: 응답시간(P95), 에러율, CPU/메모리, 비용
- 로컬보다 클라우드 측정 우선

---

## 6-1) 병목/성능 점검 체크리스트
- 프론트 ↔ 백엔드 구간 지연 확인
- 백엔드 ↔ DB 구간 지연 확인
- 특정 시간대 에러율 급증 여부
- CPU/메모리 스파이크와 응답시간 상관관계

---

## 7) 제품 단계 순서
1. BM (수익 구조)
2. PoC (가능성 검증)
3. MVP (최소 기능 출시)
4. PMF (시장 적합성 확보)
→ BM 없이 MVP 금지

---

## 8) 자동화 원칙
- `claude -p` + 스케줄러로 자동 코드 리뷰 가능
- permission-mode는 안전모드 우선
- 반복 작업은 커맨드/스니펫화

---

## 8-1) 자동화 루틴 예시
- 매일 23:00 코드 품질 체크
- 매주 월요일 아침 주간 리포트 생성
- 결과는 JSON으로 저장 후 리뷰만 확인

---

## 9) 앞으로의 코딩 규칙 (요약)
- 계획 없는 구현 금지
- 범위 추가 즉시 비목표로 분리
- 결제/보안/모니터링은 항상 우선순위
- 마일스톤 단위로 끊고 검증 후 커밋

---
